# セルフホスト型 RSS リーダー 機能提案書

作成日: 2026-07-06 / 対象: rss-reader（Rust/Axum + SolidJS + Claude API、個人〜家庭用セルフホスト）

---

## 1. 現状サマリ

本プロダクトは既に「個人用 AI リーダー」として商用サービスに比肩する完成度にある。25 のバックエンドスライス・22 本のマイグレーションで、以下が**実装済み**である。

**強み（市場比較で優位な領域）**

- **AI 統合の深さ**: オンデマンド要約・翻訳（DB キャッシュでトークン節約、モデル/プロンプト個別設定）、記事 Q&A（Ask Claude）、AI タグ提案、関連度スコアリング（興味プロファイル採点＋重要順ソート）、日次ダイジェスト、話題クラスタリング＋統合要約。Inoreader Intelligence / Feedly Leo が有料上位ティアで売る機能群を、自前 API キー・オンデマンド呼び出しで固定費ゼロ運用できる。これは OSS セルフホスト勢（Miniflux/FreshRSS）に対する最大の差別化。
- **TTS リッスンモード**: 本文/要約/翻訳のソース切替読み上げ、発音辞書（組込 147 語＋ユーザー辞書）、位置永続化、読了 80% での既読合流。Web Speech ベースでトークン消費ゼロ。
- **PWA / レスポンシブ**: 3 ペイン⇆モバイル master-detail、installable PWA、VAPID Web Push（純 Rust、優先フィード新着通知）。
- **読書体験**: readability 全文抽出、pg_trgm 全文検索、滞在/スクロール起点の既読化、キーボードナビ、4 テーマ、スター＋ハイライト/注釈。
- **整理・自動化**: フォルダ、タグ、スマートビュー、If/Then ルールエンジン、ミュート、OPML 入出力、フィード自動発見（ページ URL からの RSS 検出）、Instapaper 連携、フィード健全性監視。
- **堅牢性**: Argon2id + Cookie セッション認証、SSRF 三重防御、NDJSON エクスポート/インポート、Vertical Slice で拡張コストが低い。

**既知の弱点（インベントリの gaps より）**: クローラが直列・条件付き GET なし、digest メール送信がスタブ、実装済み機能への UI 導線不足（/digest・/clusters・横断 Ask・保存 Q&A・後で読む一覧・タグ/スター絞り込み）、記事の無期限蓄積、一覧のページネーション無し。加えて、**pg_dump スケジュールバックアップは既定オフかつランタイムイメージに pg_dump が無く不動作疑い**（tech-debt レビュー D-3、HIGH）— 上記 NDJSON 入出力は手動操作前提であり、無人バックアップは現状実質未整備である。

## 3. ギャップ分析 — 市場では標準だが本プロダクトに無いもの

1. **条件付き GET（ETag / If-Modified-Since / 304）とアダプティブポーリング** — Miniflux/FreshRSS を含む全リーダーの標準。現状は毎サイクル全フィードを直列フル取得しており、配信元にもサーバーにも無駄が大きい。docs/review の機能リサーチでもトップピック #1。
2. **記事保持ポリシー** — 現状無期限蓄積。Miniflux の保持設定、NewsBlur ですら「アーカイブは上位プランの売り」であり、無制限保持は意図的な機能であって既定ではない。ページネーション無しの一覧全件フェッチと相まって、時間とともに確実に劣化する。
3. **読書フロー内のテキスト選択 AI アクション** — Readwise Ghostreader / Matter Co-Reader が確立した 2025-26 の実装パターン。「読んでいる場所で選択→質問/翻訳/平易化」。本プロダクトは Ask がチャットパネル固定で、選択起点の動線が無い。
4. **複数記事の統合レポート** — Feedly AI Actions / Inoreader 自動レポートの中核機能。バックエンド（`askArticles`）は実装済みだが**画面が無い**。
5. **ハイライト・ノートの PKM 還流（Markdown/Obsidian エクスポート）** — Instapaper/Wallabag/Readwise で標準要件化。溜めたハイライトの出口が現状無い。
6. **任意 URL の手動保存（read-it-later クリップ）** — Readwise Reader・Instapaper・Wallabag の中核機能。「フィード外で出会ったページを保存して読む」導線が本プロダクトには無く、read-later 系との比較で最も目立つ欠落。
7. **ニュースレター → フィード化** — Feedbin/Feedly/Inoreader が標準装備、Kill the Newsletter が OSS 定番。設計書 30 がスタブのまま。
8. **RSS を持たないページのフィード化** — FreshRSS（XPath/CSS）と Inoreader（Web feeds）の看板機能。
9. **ルールエンジンへの AI アクション接続** — Inoreader は 2025 年にルールのアクションへ「要約作成・翻訳」を追加。本プロダクトはルール基盤も LLM 基盤も持ちながら接続されていない。
10. **一覧での重複記事集約** — Feedly Leo のデデュープ、Inoreader 重複フィルタ。クラスタリング実装済みなのに一覧表示に反映されていない。
11. **サードパーティクライアント API（Google Reader/Fever 互換）** — OSS 5 製品すべてが対応するデファクト。設計 29 で見送り済みだが、市場ではリーダー選定条件になっている。
12. **プライバシー保護（トラッカー除去・トラッキングパラメータ削除・画像プロキシ）** — Miniflux がセルフホストの基本価値として定着させた領域。

## 4. 機能提案（優先度順）

### P1 — 必要（基礎体力と「作った資産の回収」）

#### P1-1. 条件付き GET + アダプティブポーリング（apalis 移行に織り込む）
- **概要**: フィード取得に ETag/Last-Modified を保存して `If-None-Match`/`If-Modified-Since` を送り 304 ならスキップ。あわせて投稿頻度に応じた per-feed 取得間隔と失敗時バックオフ。着手中の apalis 移行（cron + Postgres ジョブキュー）の要件に含めて一括実装する。
- **市場根拠**: 全競合の標準。Miniflux は「地味だが確実」なフェッチャ改善を 2025 年も継続。直列ループはインベントリでも技術負債 B-5 として特定済み。
- **規模**: M（apalis 移行込みで L）。`feeds/repository.rs` に etag/last_modified 列（migration 追記）、`fetch_and_store` の入口で条件付きヘッダ付与。既存の唯一のクロール集約点に閉じるため親和性は高い。
- **置き場所**: `shared/scheduler.rs` の差し替え + `features/feeds/`（既存スライス内の取得系変更で完結）。

#### P1-2. 既存 AI 資産の導線・接続の完成（発見性スプリント）
- **概要**: ①サイドバーに /digest・/clusters リンク ②実装済み API `askArticles` の UI（記事を複数選択→「Claude に横断質問」→レポート生成。Feedly AI Actions 相当）③保存済み Q&A ノートの閲覧 UI（`getArticleNotes`）④タグ・スターでの一覧絞り込み ⑤「後で読む」一覧（`listReadLater`）。
- **市場根拠**: クロス記事レポートは Feedly の高額 B2B プランの売り、Inoreader Pro の目玉。本プロダクトはバックエンド実装済みで、**競合の課金機能が UI 数枚の距離にある**。機能が発見不能では実装コストが回収できない。
- **規模**: S〜M。バックエンド変更ほぼゼロ（タグ/スター絞り込みは articles の一覧クエリにパラメータ追加程度）。
- **置き場所**: フロントのみ中心（`SidebarContent.tsx`・新規 `components/article/CrossAsk.tsx` 等）。絞り込みは `features/articles/` の list ハンドラ拡張。

#### P1-3. 記事保持ポリシー（自動パージ）+ 一覧ページネーション
- **概要**: 「N 日より古い既読・非スター・非ハイライト記事を自動削除」する設定（既定 off、フィード単位 override は将来）。スケジューラに日次パージを追加。あわせて一覧 API に limit/offset（または cursor）を入れ、フロントを無限スクロール化。
- **市場根拠**: Miniflux/FreshRSS/Inoreader が標準装備。「無期限アーカイブ」は NewsBlur が有料の売りにするほどで、既定で無制限蓄積は運用リスク。機能リサーチのトップピックにも掲載。
- **規模**: M。スター・ハイライト・ノート参照の保護条件が肝（FK は整備済み）。
- **置き場所**: 新スライス `features/retention/`（設定 CRUD + パージ実行）+ `shared/scheduler.rs` にループ 1 本。ページネーションは `features/articles/` 内。

#### P1-4. テキスト選択 AI アクション
- **概要**: 本文で範囲選択すると小さなポップオーバーが出て「質問する / 翻訳 / 用語をわかりやすく / ハイライト保存」。質問系は選択文を引用として既存 Ask チャットに注入して開く。ハイライト保存は既存 annotations に接続。
- **市場根拠**: Readwise Ghostreader・Matter Co-Reader が確立した「読書フローを離れない AI」パターンで、2025-26 の read-later 系の共通解。機能リサーチのトップピックでも裏付け済み。オンデマンド AI という本プロダクトの思想に最も合致する（ユーザーの明示操作でのみトークン消費）。
- **規模**: S〜M。バックエンドは既存 `ask` スライスをそのまま使い、新 API 不要。フロントの selection ポップオーバーが主作業。
- **置き場所**: フロント `components/article/SelectionMenu.tsx` + 既存 `ArticleAsk` / `Annotations` への接続。

#### P1-5. digest メール送信の決着（実装 or 設定撤去）
- **概要**: `digest/email.rs` のスタブを lettre で実装するか、SMTP 設定 6 項目ごと削除するかを決める。**推奨は実装**（S 規模）: ダイジェスト生成は完成済みで、メール配信は「朝に届く」という消費文脈を足すだけ。
- **市場根拠**: 「要約プローズ＋スケジュール配信＋重複排除」が 2026 年の AI ダイジェストの基準（Readless 等）。Inoreader もメールダイジェストを Pro に同梱。生成だけあって配信が無い現状は基準の 2/3 で止まっている。
- **規模**: S（lettre 導入・送信・失敗ログ。リトライは既存方針どおりログのみで可）。
- **置き場所**: `features/digest/email.rs` の差し替えのみ。スライス外に波及しない。

### P2 — 便利（差別化を伸ばす）

#### P2-1. ルールエンジンへの AI アクション接続
- **概要**: 既存 automation_rules のアクションに「要約を生成」「翻訳を生成」「AI 判定でタグ付け」を追加。暴走防止に「優先度高フィード限定」「1 サイクル N 件上限」のガードを必須にする。
- **市場根拠**: Inoreader が 2025 年 10 月にまさにこの拡張（ルール→要約/翻訳アクション）を行い乗り換え理由の筆頭と評される。本プロダクトはルール基盤・LLM 基盤・キャッシュ列がすべて既存で、接続コードだけが無い。
- **規模**: S〜M。`automation_rules/service.rs` の action 分岐に articles サービスの既存要約/翻訳ロジックを呼ぶ分岐を追加。
- **注意**: 「オンデマンド AI」原則の唯一の例外になるため、既定 off・上限付き opt-in とする。

#### P2-2. ニュースレター → RSS 取り込み（設計書 30 の詳細化・実装）
- **概要**: 宛先アドレス（例: `xxx+token@home.example`）宛のメールを受けてフィード化。自前 SMTP 受信は重いので、Cloudflare Email Routing / Mailgun inbound 等の Webhook 受信を第一候補に、購読ごとのトークン付きアドレス発行・個別無効化（Feedbin 方式）を踏襲。
- **市場根拠**: Feedbin の看板機能、Feedly Pro+ / Inoreader Pro が枠上限付きで有料提供、OSS では Kill the Newsletter が定番。「メールに分散する購読物の一元化」は商用 3 社が揃って有料機能化する検証済みニーズ。
- **セルフホスト価値観との緊張（明記）**: 本書は「提案しない機能」で n8n 連携を外部 SaaS ハブ前提として却下しており、外部メール受信サービスの採用はその判断と緊張関係にある。ただしメール受信は到達性・スパム対策・逆引き等の理由で個人の自前 SMTP 運用が現実的でない固有領域であり、次の条件で例外を許容する: ①外部依存を inbound 受信経路 1 点に限定し、コア機能はそれ無しで完全動作 ②受信は汎用 Webhook 形式とし提供者を差し替え可能に ③完全自己ホスト代替経路（受信専用 SMTP の Maddy 等、または Kill the Newsletter 自己ホスト連携）を設計書 30 に併記。この条件を満たす設計にできない場合は P3 に降格する。
- **規模**: L（受信経路の外部依存設計を含むため、着手前に設計書 30 を詳細化）。
- **置き場所**: 新スライス `features/newsletter/`（アドレス CRUD + inbound webhook ハンドラ + 仮想フィードへの記事 upsert）。articles テーブルを再利用できるかが設計論点。

#### P2-3. 一覧での重複記事集約表示
- **概要**: クラスタリングの重複判定結果を記事一覧に反映し、「同一話題 n 件」をまとめて 1 行＋バッジで表示（展開可能）。
- **市場根拠**: Feedly Leo のデデュープ、Inoreader 重複フィルタ、Readless の「6 フィードの同じ発表を 1 件に統合」— 2026 年のトリアージ標準。バックエンド（trigram クラスタ・重複判定・LLM 不使用）は完成済みで、一覧への接続だけが欠けている。
- **規模**: M。一覧 API にクラスタ ID を同梱するか、フロントでクラスタ情報を突合するかの設計判断あり。
- **置き場所**: `features/articles/` の一覧 read model 拡張 + `ArticleList.tsx`。

#### P2-4. ハイライト・ノートの Markdown エクスポート（Obsidian 還流）
- **概要**: スター/ハイライト/Q&A ノートを記事単位・期間単位で Markdown（YAML frontmatter 付き）としてエクスポート。まずは「クリップボードへコピー / .md ダウンロード」で十分。
- **市場根拠**: Instapaper の Notion/Obsidian 連携、Wallabag の注釈 API が「セルフホスト版 Readwise」構成の核と評される。PKM 還流は標準要件化しており、出口の無いハイライトは溜まらない。
- **規模**: S。既存 annotations / ask リポジトリの読み出し＋整形のみ。
- **置き場所**: `features/annotations/` に export ハンドラ追加（新スライスを切るほどではない）。

#### P2-5. 読了時間表示 + コマンドパレット（読書 UX クイックウィン 2 点）
- **概要**: ①一覧・詳細に推定読了時間（文字数÷日本語 500-600 字/分、フロント計算のみ）②Cmd+K でフィード移動・記事検索・アクション（既読化・スター・要約生成）を横断実行するパレット。既存の keyboard.ts / Ark UI 資産を流用。
- **市場根拠**: どちらも機能リサーチのトップピックで、読了時間は「極小コストなのに無い」筆頭。コマンドパレットは Readwise Reader 等モダンリーダーの標準操作系。
- **規模**: 読了時間 S、パレット M。バックエンド変更なし。
- **置き場所**: フロントのみ（`lib/reading-time.ts` 新設、`components/CommandPalette.tsx`）。

#### P2-6. 一覧の見出し一括翻訳
- **概要**: フォルダ/フィード単位の明示ボタンで、一覧に表示中の外国語記事タイトルをバッチ 1 リクエストでまとめて翻訳し、一覧に原題併記で表示。結果は記事単位でキャッシュし再消費しない。
- **市場根拠**: Folo が定着させた「一覧段階での言語バリア除去」で、機能リサーチのトップピック #6。本文翻訳（実装済み）より手前の「読むか決める」段階を助ける。
- **オンデマンド原則との整合**: 自動翻訳にはせず、ユーザーの明示操作＋表示中件数のみを対象とすることで原則内に収める（1 回の消費もタイトル数十件分と小さい）。
- **規模**: S〜M。既存 LLM 基盤・キャッシュ方針を流用。バッチプロンプトと一覧 UI の併記表示が主作業。
- **置き場所**: `features/articles/` にバッチ翻訳ハンドラ + `ArticleList.tsx`。

#### P2-7. 任意 URL の手動保存（read-it-later クリップ）
- **概要**: フィード外の任意 URL を保存して本文抽出し、「クリップ」仮想フィードの記事として読む。入口は ①設定/サイドバーの URL 入力 ②PWA manifest の `share_target`（iOS/Android の共有シートから直接保存）。保存後は既存の読書体験（要約・翻訳・TTS・ハイライト・後で読む）がそのまま使える。
- **市場根拠**: Readwise Reader・Instapaper・Wallabag の中核機能で、「Web で出会った長文を一元管理する」ニーズは read-later カテゴリの存在理由そのもの。本プロダクトは extraction（readability）・safe_get（SSRF ガード）・PWA 基盤がすべて既存のため、競合の中核機能を低コストで獲得できる。
- **規模**: M。取得→抽出→articles への upsert は既存部品の組み合わせ。share_target のルーティングとクリップ用仮想フィードの扱いが設計点。
- **置き場所**: 新スライス `features/clips/` + manifest への `share_target` 追記 + サイドバー導線。

#### P2-8. Web ページのフィード化（CSS セレクタ・スクレイプフィード）
- **概要**: RSS を出さないページを URL + CSS セレクタ（記事リンク・タイトル）指定で仮想フィード化し、通常のクロールサイクルで差分検出。SSRF ガード（safe_get）とサイズ上限は既存基盤をそのまま使う。実装済みのフィード自動発見（`features/feed_discovery/`）が「RSS が存在するページを見つける」機能であるのに対し、こちらは「RSS が存在しないページを購読する」機能で、役割が補完的。
- **市場根拠**: FreshRSS 独自の強みとして頻繁に言及され、Inoreader Web feeds / Feedly RSS Builder は有料の売り。「RSS が無い情報源」問題はブラウザの RSS 検出廃止以降むしろ悪化している。
- **規模**: M〜L（セレクタ UI とパース失敗時の運用が重い）。
- **置き場所**: 新スライス `features/scrape_feeds/` + feeds のクロール経路に種別分岐 1 点。

### P3 — 野心的（条件付きで着手）

#### P3-1. Google Reader API 互換（設計 29 の再評価）
- **概要**: Reeder 等のネイティブクライアントから接続できる互換 API 層。
- **市場根拠**: Miniflux/FreshRSS/TT-RSS/Yarr/CommaFeed 全対応のデファクトで、市場調査は「サーバー選定条件」とまで評価。一方で本プロダクトは自前フロント（PWA/TTS/AI）に価値が集中しており、外部クライアントからは AI 機能が使えない。**「iOS ネイティブアプリで読みたい」欲求が実際に生じたときのみ着手**（見送り判断の維持を推奨、市場根拠は再評価に足るためスタブは残す）。
- **規模**: L。新スライス `features/greader_api/` + 独自トークン認証の併設。

#### P3-2. AI 音声デイリーブリーフ（digest × TTS の合流）
- **概要**: 日次ダイジェストを「読み上げ最適化文体」で生成する派生プロンプトを足し、朝ワンタップで listen する導線（ListenBar 再利用）。
- **市場根拠**: Inoreader のプレイリスト・Readwise の全文書 TTS 等「聴く消費」がカテゴリ標準化。digest と TTS が両方完成済みの本プロダクトでは合流点 1 つで成立する。機能リサーチのトップピックにも掲載。
- **規模**: S〜M（本命 AivisSpeech 自己ホスト音声は別件の保留ロードマップに従う）。
- **置き場所**: `features/digest/` にブリーフ variant + フロント Digest 画面に ListenBar 接続。

#### P3-3. プライバシー強化（トラッキングパラメータ除去・トラッカー画像除去）
- **概要**: 保存時に `utm_*` 等の除去、既知トラッキングピクセルの sanitize 段階での削除。画像プロキシは第 2 段階。
- **市場根拠**: Miniflux が「セルフホストの基本価値」として定着させた領域で、本プロダクトの利用動機（追跡されない読書）と整合する。
- **規模**: パラメータ/ピクセル除去は S（`lib/sanitize.ts` + 取り込み時正規化）、画像プロキシは M（新スライス + CSP 調整）。

#### P3-4. YouTube / Podcast ソース対応
- **概要**: YouTube チャンネル・ポッドキャスト RSS を購読し、enclosure をプレイヤー付きで表示。文字起こし→要約は Claude 接続の将来枠。
- **市場根拠**: 「全情報源の統合タイムライン」化（Reeder/Inoreader/Folo）が新世代標準。YouTube 自体が 2025 年に RSS を配信インフラとして再採用。
- **規模**: 表示対応は M（feed-rs は enclosure を解釈可能）、文字起こしまで含めると L。まず表示対応のみ切り出すのが現実的。

### 実装順の推奨

P1-2（S、即効・バックエンド不要）→ P1-5(S) → P1-4(S〜M) → P2-4/P2-5/P2-6 のクイックウィン → P1-1 + P1-3（apalis 移行と運用基盤、まとまった工数）→ P2-1/P2-3（AI 接続）→ P2-7 → P2-2/P2-8（ソース拡張、要設計書詳細化）。あわせて、弱点に挙げたスケジュールバックアップ不動作疑い（D-3）は新機能に先行して決着させることを推奨（機能追加ではなく既存負債のため本提案の番号外）。

## 5. 提案しない機能

| 市場にある機能 | 提案しない理由 |
|---|---|
| **マルチユーザー / チーム共有・権限**（FreshRSS、Feedly B2B） | 単一ユーザー前提が認証（シングルトン credential）・LoginLimiter・バックアップまで一貫した設計前提。家庭内共有は現状の単一アカウント同時ログイン（セッション複数）で足りる。 |
| **AI Feeds / 大規模キーワード監視**（Feedly の 1000+ AI モデル、5 万ソースバンドル） | クラウド側の巨大なクロールインフラが前提の B2B インテリジェンス機能。個人の購読フィード範囲ならスマートビュー＋ルール＋関連度スコアで代替済み。 |
| **常時学習型レコメンド**（NewsBlur Intelligence Trainer の常時分類、Folo の推薦） | 本プロダクトは「オンデマンド AI・挙動が予測可能」が思想。全記事を裏で継続採点する構成はトークン費用と透明性の両面で不適。既存の明示ボタン式 relevance が同じ課題への正解。 |
| **クラウド HD 音声 TTS**（Matter、Instapaper AI Voices、Readwise） | TTS 自然化ロードマップで「クラウド TTS 不採用・本命は AivisSpeech 自己ホスト」と決定済み。従量課金の音声 API はセルフホストの費用ゼロ運用と矛盾する。 |
| **トークン従量制・フリーミアム課金設計**（Inoreader 月 100 万トークン等） | 自前 API キーの持ち込みモデルであり課金メーターは不要。参考にすべきは課金でなく「使用量の可視化」のみ（技術負債②の LLM 基盤整備で扱うのが適切）。 |
| **間隔反復クイズ・忘却曲線ダイジェスト**(Recall、Readwise Daily Review) | 学習アプリの領分で、ニュース消費が主目的の本プロダクトではハイライトの Obsidian 還流（P2-4）で PKM 側に委ねる方が筋が良い。 |
| **n8n 等の汎用オートメーションハブ連携**（Inoreader） | 外部 SaaS ハブ前提。個人セルフホストなら将来 Webhook 1 本（ntfy 等）で足り、優先度は既存 Web Push が既にある分さらに低い。なお P2-2 のニュースレター受信も外部サービスに触れるが、あちらは「受信経路 1 点限定＋差し替え可能＋自己ホスト代替併記」を条件とした例外である（P2-2 参照）。 |
| **Fediverse / Bluesky 双方向ブリッジ**（Bridgy Fed 等） | 「読む」を超えた投稿・リアクション機能はリーダーの責務外。読む側だけなら Bluesky プロフィールの公開 RSS を通常フィードとして購読すれば今日から可能で、専用アダプタ不要。 |
| **公開インスタンス / 広告 / ソーシャル機能** | セルフホスト・プライベート・「読むことだけに最適化」という Miniflux 的価値観と正面から矛盾。市場でも商用リーダーの広告挿入への反発がセルフホスト回帰の動因になっている。 |

---

**総括**: 次の一手は新機能の増築ではなく、(a) クローラと保持ポリシーという「セルフホストの基礎体力」を市場標準に引き上げること、(b) 既に実装済みの AI 資産（横断 Ask・クラスタ・ダイジェスト・タグ）を UI 導線とルール接続で回収すること。この 2 つで「Folo 以外に競合のいない AI × セルフホスト」ポジションが名実ともに固まる。

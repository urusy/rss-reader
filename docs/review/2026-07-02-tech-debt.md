## A. セキュリティ(2026-07-02 監査済み・未対応)

> 5監査(SQLi/XSS/依存/並行性/構成)で実在検証済み。SQLi ゼロ・XSS 多層防御・openssl 排除・レースフリーは確認済みで、実害は以下に集約。

1. **🔴 SSRF** — `feeds`/`feed_discovery`/`extraction` の URL 検証が scheme 判定のみ。reqwest が最大10リダイレクト追従、既定で `AUTH_TOKEN` 未設定なら `/api` 全開放。private/loopback IP の拒否+各リダイレクトホップの再検証+`AppError::Upstream` のエラー文字列マスキング(`error.rs:39` が内部ポートスキャンのオラクルになる)。Miniflux 2.2 / CommaFeed 7 の「既定ブロック+opt-in」が定石。
2. **🟠 フィード本文サイズ無制限** — `feeds/service.rs:117-119` の `.bytes()` が無制限で OOM→コンテナ全死。
3. **🟠 通知取りこぼし** — `notifications/service.rs:79-97`。1サイクル>20件で `.take(20)` なのにウォーターマークを `until` まで進め、21件目以降が永久スキップ。
4. **🟠 Push 直列インライン送信** — クロールタスク内で全購読を直列送信。死んだ購読で更新が数分ウェッジ。
5. **🟠 `CorsLayer::permissive()`**(`features/mod.rs:75`)をフロントオリジンに限定。
6. 設定系: nginx セキュリティヘッダ/CSP 皆無、Postgres 既定 `rss/rss`、`cargo audit`(rsa Marvin は VAPID ES256 で未実行のため ignore 運用)。
7. LOW: ask 入力無制限、sanitize.ts の rel=noopener、sw.js push URL 未検証、send-before-commit 重複、gen-vapid.sh dd 固定 skip。

## B. バックエンド(Rust)

### B-1. [HIGH] LlmClient trait の形骸化と `llm_client` 生成の6重複
`articles/service.rs:72`, `digest/service.rs:12`, `relevance/service.rs:17`, `clustering/service.rs:16`, `tags/service.rs:13`, `ask/service.rs:12`

6つの service がほぼ同一の「APIキー取得→`AnthropicClient::new`」を各自実装し、いずれも**具象型を直接構築**。`AppState` は `LlmClient` trait を保持せず、trait に依存する service がゼロ。CLAUDE.md が trait を置く唯一の理由とする「差し替え・モック」が実装で満たされておらず、プロバイダ変更=6ファイル改修、service のモックテスト=不可能。
**方向**: 生成を `shared/llm` の1関数に集約。さらに `AppState` に `Arc<dyn LlmClient>` を持たせて service を trait 依存にするかは設計判断(要相談)。

### B-2. [HIGH] LLM 呼び出しの堅牢性ゼロ(リトライ/タイムアウト/入力上限/コスト計測)
`shared/llm/anthropic.rs:44-95`, `main.rs:29-33`

- フィード取得と共用の reqwest クライアント(**timeout 30秒固定**)を使用。生成が30秒を超えると `Upstream` で丸ごと失敗。
- 429 / 529(overloaded) への指数バックオフ・リトライなし。
- `ai_input` は full_content 全文を平文化して丸投げ(入力上限なし→長大記事でトークン超過 or コスト膨張)。
- レスポンス `usage` のログなし=コスト可観測性ゼロ。

**方向**: LLM 専用クライアント(長 timeout)分離、429/529 限定の数回リトライ、入力文字数上限、`usage` を tracing 記録。

### B-3. [MEDIUM] 通知ディスパッチの N+1(記事ごとに購読全件 SELECT+VAPID 鍵再デコード)
`notifications/service.rs:81-87,110,115` — 新着1件ごとに `broadcast()` が購読全件クエリ+P-256 鍵 base64url デコードを反復(最大20回/サイクル)。送信も完全直列。
**方向**: 購読一覧と鍵をサイクル冒頭で1回用意、送信は並行化。A-4 と同時修正が自然。

### B-4. [MEDIUM] extraction のサイズ上限が「ダウンロード完了後」判定でメモリ保護にならない
`extraction/service.rs:64-70` — `resp.bytes().await` で全量バッファ後に `extract_max_bytes` 判定。
**方向**: `Content-Length` 事前チェック+ストリーミング読みで超過時打ち切り。A-2(feeds 側)と共通ヘルパ化。

### B-5. [MEDIUM] スケジューラの単一直列ループ
`shared/scheduler.rs:19-43`, `feeds/service.rs:68-77` — 全フィードを逐次 fetch。遅い1本が全体を遅延させ、サイクル超過時は `MissedTickBehavior::Skip` でティック落ち。per-feed リトライ/バックオフなし。ミュート適用と Push 評価もインライン実行で上乗せ。
**方向**: apalis 移行(着手中・方式B)で per-feed ジョブ化+並行度制限+バックオフ。暫定は `buffer_unordered`。条件付き GET(ETag/If-Modified-Since)・アダプティブポーリング未実装も同根(→機能リサーチ文書参照)。

### B-6. [MEDIUM] service 層のテストが皆無
domain 純関数は 200 件超で良質だが、service(LLM オーケストレーション、キャッシュ判定、ウォーターマーク、ルール適用)のテストゼロ。repository テストは全て `#[ignore = "requires DATABASE_URL"]` で CI 実質未実行。B-1 の trait 形骸化がモック検証の道を塞いでいる。
**方向**: trait DI 導入後にモック LLM で service を単体化。repository は testcontainers 等で CI 実行可能に。

### B-7. [MEDIUM] ダイジェストメール送信がスタブなのに SMTP 設定6項目が config に露出
`digest/email.rs:10-31`, `shared/config.rs:38-43,114-130` — `maybe_send` はログを出すだけで実送信しない(`lettre` 依存もなし)。「設定すれば届く」という誤解を与える死に設定。
**方向**: 実装するか、config/`.env.example` から除去 or 未実装と明記。

### B-8. [LOW] その他
- **書き込み N+1**: `clustering/repository.rs:103-133`・`relevance/repository.rs:93-114` が1行ずつ INSERT(最大500件規模)。→ `UNNEST` 一括化。
- **読み取り N+1**: `automation_rules/service.rs:172-176` が記事ごとに `tags_for` を SELECT(最大5000回/apply_all)。→ `ANY($1)` でまとめ取り、tag 条件なしならスキップ。
- **インデックス欠落**: `articles.created_at`(通知の新着抽出クエリが範囲+ソートで使用)。→ 新マイグレーションで追加。
- **二重パース**: `extraction/domain.rs:151-165` の `html_to_plain_text` が HTML を2回パース、`pick_main_html`(60-76)は全 div/section 走査で準 O(n²)。
- **ハードコード**: DB プール `max_connections(10)`(`shared/db.rs:6`)、VAPID subject が `example.com` 固定(`notifications/service.rs:18`)。→ `AppConfig` へ。

### 確認済みで健全な点(誤検出防止のため明記)
認証の定数時間比較(`shared/auth.rs:54`)、エラー→HTTP 変換の一貫性、トランザクション境界(replace_clusters/save_scores/tags)、clustering union-find の unwrap は不変条件保証済み、依存構成(openssl 排除済み)。

## C. フロントエンド(SolidJS)

### C-1. [HIGH] 取得失敗が「永久ローディング」または「偽の空表示」になる
`ArticleDetail.tsx:262-265`, `ArticleList.tsx:153-160`, `Clusters.tsx`, `Rules.tsx`, `MuteRulesManager.tsx`, `FeedManage.tsx`

主要 `createResource` が `.error` を一切見ていない。記事取得 500/404 → 永久「読み込み中…」、一覧取得失敗 → 「記事がありません。」の誤表示。リトライ手段なし。
**方向**: `resource.error` を分岐する共通 `<AsyncBoundary>`+最外殻 `<ErrorBoundary>`。

### C-2. [HIGH] ListenBar の createEffect 4本に「登録順で正しさが決まる」暗黙依存
`ListenBar.tsx:155-196` — 「並べ替え禁止」コメント付きの順序依存。コンパイラもテストも保証せず、リファクタで進捗バー・既読化位置が壊れる。非リアクティブ変数3つ+signal+effect 4本の結合で認知負荷大。
**方向**: 再生状態を単一 store に集約し明示的な遷移関数へ。ロジックを tts コントローラ側へ寄せテスト可能に。

### C-3. [HIGH] fetch にタイムアウト・中断がない
`lib/api.ts:335-356` — `http<T>` が AbortSignal 非対応。LLM 系は nginx `proxy_read_timeout 120s` まで宙吊り。記事高速切替でも前リクエストが走り続ける。
**方向**: `http` に signal を通し createResource の `info.signal` を接続。LLM 系は `AbortSignal.timeout(90_000)` 等。

### C-4. [MEDIUM]
- **未読バッジ不整合**: 自動既読(`markReadNow`)が `stats` を再取得せず、行は既読でもバッジ件数が減らない(`ArticleList.tsx:50-52`, `ArticleDetail.tsx:92-100`)。→ 未読数を store 派生値に。
- **ルート分割なし**: `index.tsx:6-14` が全ルート静的 import。`marked`(Digest 専用)まで初期バンドル同梱。→ `lazy()`+動的 import。
- **alert/confirm/prompt 依存**: 失敗通知・削除確認・改名入力がブロッキングなネイティブダイアログ(ArticleDetail/FeedManage/Annotations ほか多数)。要約削除だけ Ark UI Dialog で二重基準。→ トースト+Dialog 共通ラッパに統一。
- **LLM エラーマッピング5重複**: 「503→キー未設定/502→失敗」分岐が `ArticleList`/`ArticleAsk`/`TagEditor`/`Clusters`/`Digest` にコピペ。→ ヘルパ集約。
- **StarToggle の N+1**: 記事を開くたび全 `/api/stars` を取得(`Annotations.tsx:10-11`)。記事オープン時に5リクエスト並行発火。→ store 共有リソース化。
- **ボタン外観の手書き重複**: `Settings.tsx:289-294,347-352` が `buttonVariants` 不使用、`border-input` はここだけ。→ `buttonVariants({variant:"outline"})`。
- **コンポーネントテスト空白**: テストは lib/ 純関数+ListenBar に集中。store・自動既読・楽観更新・ルート群が未検証(`@solidjs/testing-library` 導入済みで書ける状態)。

### C-5. [LOW]
- `http<T>` のランタイム検証なし+`errorStatus` が例外メッセージの正規表現パース(stringly-typed)。→ `ApiError {status, body}` クラス化。
- 記事一覧/検索結果の仮想化なし(`<For>` 素朴描画)。
- `relevanceScores` を起動時に無条件フェッチ(`store.tsx:63-66`)。
- a11y 小穴: ListenBar 進捗バーに `role="progressbar"` なし、アイコンのみ `×` ボタンに `aria-label` なし、ArticleAsk が `<form>` 未使用。
- 小ロジック重複: `?article` 抽出が3箇所、Blob ダウンロードが2箇所。
- sw.js に `skipWaiting`/`clients.claim` なし(SW 更新遅延リスク)。
- i18n: UI 文字列は全面ハードコード日本語(現方針では許容、規模数百箇所)。

### 良好な点
strict tsconfig・型キャスト7箇所のみ・any/@ts-ignore なし。sanitize 多層防御、TTS 平文化契約、theme/auth の副作用なし import。`api.ts` 692行は大半が型定義で実害小 — リファクタ価値が高い肥大は **ArticleDetail(418行・責務過多)と ListenBar(312行・effect 結合)の2つ**。

## D. インフラ・ビルド・運用

### D-1. [HIGH] CI/CD が完全不在
`.github/` 等なし。clippy/test/typecheck/build の破綻がマージ後まで露見しない。
**方向**: GitHub Actions で `cargo clippy -D warnings`+`cargo test`+`pnpm typecheck`/`test`+両イメージ build を PR トリガで。

### D-2. [HIGH] マイグレーションが起動時自動適用のみでロールバック不能
`shared/db.rs` の `sqlx::migrate!` が無条件適用、down スクリプトゼロ。破壊的変更混入時に再起動=即適用で切り戻せない(0001-0020 の採番整合は問題なし)。
**方向**: 適用前 pg_dump の手順化(D-3 とセット)。破壊的変更のみ reversible migration。

### D-3. [HIGH] DB バックアップが既定オフ+有効化しても動かない疑い
データは `pgdata` ボリューム一箇所のみ。`BACKUP_*` は `.env.example` で全コメントアウト、かつランタイムイメージ(`debian:bookworm-slim`)に **pg_dump が入っておらず**有効化しても機能しない疑い。ボリューム消失=全損(キャッシュ済み要約/翻訳=消費トークンも消える)。
**方向**: ランタイムに `postgresql-client` 追加+既定オン推奨 or README にボリュームバックアップ手順明記。**要動作確認**。

### D-4. [HIGH] マルチアーキビルド非対応
`scripts/build.sh`/`push-images.sh` に `--platform`/buildx なし。Apple Silicon 開発機→amd64 NAS で `exec format error` の典型構成。
**方向**: `docker buildx build --platform linux/amd64,linux/arm64 --push` に統合(スクリプト重複解消も同時に)。

### D-5. [MEDIUM]
- **backend healthcheck なし**: frontend の `depends_on` が起動順のみ保証、マイグレーション中に nginx が 502 を返す時間帯。→ `/api/health` を healthcheck 化(ランタイムに wget 追加 or バイナリのサブコマンド)。
- **ログローテーション・リソース制限なし**: `RUST_LOG=debug` 既定で肥大、mem 制限なしで OOM(A-2)がホストを巻き込む。→ logging max-size/mem_limit、既定 info。
- **frontend Dockerfile が `--frozen-lockfile` なし**(lockfile 未COPY と同時修正)。
- **ツールバージョン固定不完全**: rust-toolchain.toml/.node-version なし、Dockerfile `rust:1.90` と Cargo.toml MSRV 1.82 のズレ、sqlx-cli 非固定。
- **ベースイメージ更新方針なし**: ダイジェスト固定も Dependabot/Renovate もなし。

### D-6. [LOW]
- scripts の変数・イメージ名組み立てが build/push で重複。
- README 鮮度: `gen-vapid.sh`・`scripts/test/`(統合テスト24本)未言及、API 一覧が初期機能のまま(search/backup/push/tags 等の後発20スライス未反映)。
- 監視・異常通知なし: スケジューラ(digest/clustering/backup/push)失敗に気づく手段が `docker compose logs` 目視のみ。実装済み Web Push を自分宛て運用アラートに流用する軽量案あり。

## E. 推奨バッチ(まとめ方の提案)

1. **セキュリティ即応ブランチ**: A-1 SSRF + A-2/B-4 サイズ上限(共通ヘルパ) + A-3 通知ウォーターマーク(TDD、監査時の合意どおり)
2. **LLM 基盤整備**: B-1 生成集約(+trait DI 判断) + B-2 リトライ/タイムアウト/上限/usage ログ + C-4 エラーマッピング集約 — AI 機能が今後も増える前提の投資
3. **失敗時 UX**: C-1 AsyncBoundary + C-3 AbortSignal + C-4 バッジ不整合 + alert→トースト統一
4. **運用ガード**: D-1 CI + D-3 バックアップ実効化 + D-4 buildx + D-5 healthcheck/ログ上限
5. **随時**: B-8/C-5/D-6 の LOW 群、ListenBar リアーキ(C-2)は TTS Phase1(AivisSpeech)着手時に同時が効率的
